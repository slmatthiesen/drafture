# Deterministic Terraform emitter — 2026-06-30

## Idea (operator)
Today the reference Terraform is generated by an LLM from scratch per tier — a large
(~32k-token) call (`provider.generateConfig`) that's slow, the dominant API cost, and emits
"reference-only" HCL with recurring **wire-up gaps** (a CMK with no key policy, a CloudFront
origin with no cert, …). But the `design.json` is already a **typed graph**: every node is a
known `awsService` + `role` + security TAGS, and every edge is a typed payload/protocol. The
mapping *node → Terraform resources* is mostly mechanical: "we want SQS ⇒ emit this block."
So generate the HCL **deterministically from the graph**, the way costs and the security floor
are already injected deterministically — keep the LLM only for the long tail.

This is the same lever as the rest of the product (see [[stackdraft-deterministic-vs-agentic]]):
move reusable knowledge OUT of per-request LLM output and INTO the KB / code.

## Why it's worth it
- **$0 + instant** — no LLM call for any service that has a template (the expensive part today).
- **Wire-up gaps become structurally impossible.** The gaps exist because the model *forgets*
  the consequence of a resource. A template that always emits the CMK key policy alongside the
  log group, the ACM validation alongside the cert, etc. **cannot** forget. The 7 rules in
  `packages/kb/terraform-wireup-rules.json` stop being post-hoc warnings (`detectWireupGaps` in
  `apps/api/src/routes/config.ts`) and become **template invariants** — and `detectWireupGaps`
  becomes the *test* the templates must pass.
- **No hallucinated / incoherent resources**; consistent style; unit-testable with golden files.
- **Coherent with the graph** by construction (it IS the graph), so it can't contradict the
  design the way free-form HCL can.

## Architecture (provider-abstracted registry — matches our defaults)
- `apps/api/src/pipeline/terraform/` — a registry `Map<serviceKey, ServiceEmitter>` where
  `ServiceEmitter = (node, ctx) => HclBlock[]`. `serviceKey` is a normalized `awsService`
  (e.g. "lambda", "ec2", "s3", "cloudfront", "rds", "sqs", "self-managed postgres").
- `ctx` carries what an emitter needs beyond its node: `tier` (public-subnet box vs
  private-subnet+NAT differs by tier), `edges` (for SG rules + IAM wiring), the security floor,
  `region`, and a name prefix.
- An **assembler** walks `tier.nodes` → resource blocks; walks `tier.edges` → the GLUE
  (security-group rules, least-priv IAM, origin/target wiring); and prepends the always-on
  baseline (provider + variables + KMS CMK + CloudTrail + the security-floor resources).
- **Edges are the hard, valuable part.** Per-node blocks are easy; the judgment is the wiring:
  `ec2_box → postgres (localhost)` ⇒ no SG rule (same host); `cf → ec2_box` ⇒ origin + an
  ACM/EIP cert (the cloudfront-origin-tls fix); `lambda → s3` ⇒ a scoped IAM policy from the
  edge. Derive SG rules and IAM least-priv FROM the edge list + security tags.

## Hybrid fallback (coverage without boiling the ocean)
Don't require a template for every AWS service on day one. The assembler:
1. emits deterministically for any node whose `serviceKey` has a registered emitter;
2. for an unknown service, **falls back to the current LLM `generateConfig`** scoped to just
   that node's block (or flags it `# TODO: unsupported service`).
This shrinks the LLM surface from "the whole stack" to "the long tail," so most designs become
$0/instant while rare services still get covered. Track template coverage % as a metric.

## Sequence
1. Define `ServiceEmitter` + `ctx` + the assembler skeleton; port the file header/disclaimer and
   `flagIfIncomplete` from `routes/config.ts`. Golden-file test harness.
2. Implement emitters for the **budget single-box vocabulary first** (the shapes this session's
   posture change makes common): EC2 + EBS, self-managed Postgres (user-data/Compose bootstrap),
   Lambda, S3, CloudFront(+WAF, +ACM/EIP origin), CloudWatch Logs+Alarms+CMK, SNS, EventBridge
   Scheduler, Secrets Manager, CloudTrail, X-Ray. Each emits its wire-up consequence inline.
3. Edge→SG / edge→IAM derivation. Assert `detectWireupGaps()` returns ZERO on the templated
   output (the gaps are now structurally impossible) — make that a test.
4. Wire the assembler into `generateDesign.ts` (`--with-tf`) and `/api/config` behind a flag;
   keep the LLM path as the hybrid fallback. Compare against the LLM `.tf` on the dogfood packs.
5. Expand the service catalog (balanced/resilient: ALB, Fargate/ECS, RDS, ElastiCache, NAT, …);
   raise coverage %; demote the LLM path to fallback-only.

## Payoff vs cost
Big upfront code (a catalog of emitters + the edge-wiring glue), but it's **all $0 to build and
test** (deterministic — golden files, no API), it KILLS the recurring TF API cost (the
[[stackdraft-cost-discipline]] leak), and it removes the wire-up-gap class entirely. Strong fit
for the product thesis: judgment in the graph, mechanics in the KB.

## Open questions
- Inline resources vs a root module calling per-service child modules? (Start inline; modularize
  later if the catalog grows.)
- How much edge→IAM least-priv can be derived purely from the graph vs needs a per-service hint
  in the KB? (Probably a small `iamNeeds` field per emitter.)
- Keep `terraform-wireup-rules.json` as the spec the templates are tested against (yes — it
  becomes the contract, not a prompt).
